# HASI 论文代码与结果整理（2026-07-07）

本文档按当前仓库 artifacts 整理，目的是写论文时快速定位“可引用代码”和“可进入论文表格/分析的结果”。整理依据是当前本地文件，不把 `results/archive/`、旧 `non_shared_base`、旧 `previous_results` 与当前 shared-base 主线混算。

## 1. 当前论文主线

推荐论文叙事顺序：

1. **方法主线**：HASI = hub-aware selection + ERF affected region + anchor stabilization + structural inpainting + DAR + local fine-tuning。
2. **统一协议主线**：所有方法共享同一个 `results/shared_base/{dataset}/seed{seed}` 和同一个 JSON forget set；主 split 是 `stratified_random` 的 60/20/20，base 训练图是 `train_subgraph`。
3. **主证据数据集**：`pubmed` 是最稳的主证据；`primekg-disease-gene-small-nosource` 是医学 KG NoSource 主线，但隐私优势不稳定；`hetionet-small-nosource` 更适合作为 biomedical robustness 支撑。
4. **扩展对照**：同时保留 repo-native baselines 与 OpenGU-adapted baselines。OpenGU 适配结果应表述为 “adapted from OpenGU and integrated into the unified protocol”，不要写成未修改官方代码。

不要把这些当主表：`results/archive/`、`results/non_shared_base_200epoch_random_all_20260527/`、`results/previous_results_excluding_tuning_20260608/`、带 source leakage 的 `primekg-disease-gene-small`。

## 2. 有用代码索引

### 2.1 HASI 方法实现

| 路径 | 论文中可对应的内容 |
| --- | --- |
| `src/hasi/unlearner.py` | HASI 主流程。`HASIConfig` 定义 ERF、inpainting、DAR、anchor、fine-tune 等参数；`HASIUnlearner.preprocess()` 做 hub score 与 anchor 划分；`plan_node_unlearning()` 计算 affected region 和 DAR context；`unlearn_nodes()` / `unlearn_edges()` / `unlearn_features()` 是三类遗忘入口。 |
| `src/hasi/hub_identification/hub_scorer.py` | HubScore 计算。可用于方法章节解释结构中心性/梯度敏感性融合。 |
| `src/hasi/erf_partitioning/ppr.py` | ERF/PPR affected region 计算。 |
| `src/hasi/anchor_stabilization/anchor_manager.py` | primary / secondary / regular anchor 分类。 |
| `src/hasi/anchor_stabilization/anchor_loss.py` | anchor stabilization loss。 |
| `src/hasi/structural_inpainting/inpainter.py` | structural inpainting 触发和 MGAE 修复逻辑。 |
| `src/hasi/dar/pipeline.py` | DAR distributed anchor replacement。 |

## 10. 写论文时的安全边界

1. 主表优先用 `results/{dataset}_eval/aggregate_summary.csv` 和 `docs/HASI_tuned_vs_baselines_three_datasets_summary_2026-07-05.md`，不要递归聚合整个 `results/`。
2. `gif` 是 edge-only，不应出现在 node/feature baseline 主表。
3. PrimeKG-NoSource 的 node privacy 不稳定，要写成 trade-off/limitation，而不是全面优于。
4. PrimeKG disease-gene-small 旧版本有 source leakage 风险，保留为归档线索，不用于主论文结果。
5. `configs/final/main_matrix.yaml` 的范围偏旧；如果要重跑医学主线，应该按 `results/*_eval` 当前实际协议重新生成 matrix。
6. 所有 claims 尽量同时给出：`forget_set.path`、`base_artifact.path`、dataset/type/ratio/seed/forget seed，这样可以避免配对公平性质疑。
