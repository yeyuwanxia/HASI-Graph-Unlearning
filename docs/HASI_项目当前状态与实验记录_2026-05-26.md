## 4. Forget Sets 的来源与生成逻辑

forget sets 由以下脚本生成：

```text
experiments/generate_forget_sets.py
```

核心选择逻辑如下。

### 4.1 Node Forget Sets

#### random_train

`random_train` 从 `train_mask` 中随机选择节点：

```text
select_forget_nodes(data, ratio, seed, mask_name="train_mask")
```

含义：

```text
训练标签节点遗忘 benchmark
```

它适合标准 Planetoid 半监督节点分类设置下的方法对比。

注意：PubMed 的 Planetoid split 中：

```text
总节点数：19717
train_mask：60
```

因此 PubMed 中：

```text
random_train r0p05 = 3 个节点
random_train r0p1  = 6 个节点
```

这不是代码错误，而是由 `train_mask` 的大小决定的。

#### random_all

`random_all` 从全图节点中随机选择节点：

```text
select_forget_nodes(data, ratio, seed, mask_name="")
```

含义：

```text
全图随机节点删除
```

它更接近医疗图故事线中的 `random paper node deletion`。

#### hub

`hub` 根据 HubScore 从高到低选择节点：

```text
HubScorer().compute_hub_scores(graph)
```

含义：

```text
删除结构重要节点 / 医疗 hub 节点 / 高影响 paper 节点
```

当前实现中，`hub` 是排序选择，因此相同数据集和 k 下不同 seed 的 forget targets 一样。seed 仍会影响模型训练随机性。

#### low_degree

`low_degree` 根据节点度数从低到高选择节点：

```text
sorted(graph.nodes, key=(degree, node_id))
```

含义：

```text
删除结构影响较弱的低度节点
```

它是 hub deletion 的对照组。

### 4.2 Edge Forget Sets

edge forget set 从 `edge_index` 中随机选择边：

```text
select_forget_edges(data, ratio, seed)
```

它没有 `train_mask`，因为 `train_mask` 是节点分类任务中的训练节点掩码，不适用于边。

### 4.3 Feature Forget Sets

feature forget set 从特征维度中随机选择：

```text
select_forget_features(data, ratio, seed)
```

它也没有 `train_mask`，因为 feature deletion 的对象是特征维度，而不是节点。

## 6. 当前暴露出的三个主要问题

### 问题 1：不同数据集需要不同配置

当前已经认识到：HASI 的 tuned 配置不应该只有一个全局配置，而应按数据集和遗忘类型拆分。

当前已经建立：

```text
configs/tuned/by_dataset/
  cora/
    node.yaml
    edge.yaml
    feature.yaml
  citeseer/
    node.yaml
    edge.yaml
    feature.yaml
  pubmed/
    node.yaml
    edge.yaml
    feature.yaml
```

原因：

- Cora / CiteSeer 是小图，PubMed 是中等规模图。
- PubMed node/edge/feature 的最佳参数与 Cora 不完全一致。
- edge、feature 对 `forget_weight`、`inpainting`、`feature_drift_threshold`、`edge_forget_loss_mode` 等参数更敏感。

当前处理方式：

- final 主实验的 `hasi_tuned` 使用：

```text
configs/tuned/by_dataset/{dataset}/{unlearning_type}.yaml
```

后续建议：

- 不再用单个 tuned 配置覆盖所有数据集。
- 文档中应明确区分 default config 和 dataset/type-specific tuned config。

### 问题 2：random / low-degree / hub 三类对比在 PubMed 上存在数量问题

文档要求比较：

```text
random deletion
low-degree deletion
medical hub deletion
```

但如果直接使用：

```text
random_train r0p05/r0p1
hub r0p05/r0p1
low_degree r0p05/r0p1
```

会出现严重不公平：

```text
random_train：3 / 6 个节点
hub：986 / 1972 个节点
low_degree：986 / 1972 个节点
```

因此，第三步 hub deletion 不应该使用 `random_train` 对比 ratio 型 hub/low_degree。

当前推荐设计：

```text
random_all_k60 vs low_degree_k60 vs hub_k60
random_all_k120 vs low_degree_k120 vs hub_k120
```

这样三组删除数量一致，才能回答：

```text
在删除数量相同的情况下，hub deletion 是否更难？
```

### 问题 3：HASI 对比 baseline 使用的数据集与删除口径需要说明

当前 final main matrix 已经完成：

```text
Cora / CiteSeer / PubMed
node / edge / feature
hasi_default / hasi_tuned / retrain / grapheraser_bekm / grapheraser_blpa
```

但需要注意：

- node 主实验使用的是 `random_train`，即训练标签节点删除。
- 这适合 Planetoid benchmark 和方法间公平比较。
- 但它不完全等同于医疗故事线中的全图 `random paper node deletion`。

因此，写论文或报告时应区分两种实验口径：

```text
Final main benchmark:
  使用 random_train node deletion。
  目标是公平比较 HASI 与 baseline。

PubMed hub fragility / medical story:
  使用 matched-count random_all / low_degree / hub。
  目标是证明 hub deletion 比普通 random deletion 更难。
```

也就是说，主实验不是无效，但表述必须准确：

```text
node unlearning uses random training-node deletion under Planetoid splits
```

不能写成：

```text
random PubMed paper deletion over the full graph
```

## 10. HASI 与 baseline 主实验使用的遗忘节点数量

```text

retrain
grapheraser_bekm
grapheraser_blpa
```

| CiteSeer | random_train | 0.10 | 42 / 123 / 2024 | 12 | HASI default / HASI tuned / retrain / GraphEraser-BEKM / GraphEraser-BLPA |
| PubMed | random_train | 0.05 | 42 / 123 / 2024 | 3 | HASI default / HASI tuned / retrain / GraphEraser-BEKM / GraphEraser-BLPA |
| PubMed | random_train | 0.10 | 42 / 123 / 2024 | 6 | HASI default / HASI tuned / retrain / GraphEraser-BEKM / GraphEraser-BLPA |
